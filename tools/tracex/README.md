# tracex

可视化路由追踪（mtr 风格）。循环跑 traceroute，按跳累计统计：丢包率、最近/平均/p95 延迟、sparkline。

**一眼看出"卡在哪一跳"**——是出门第一跳、运营商骨干、还是目标侧。

## 用法

```bash
tracex baidu.com               # 默认无限循环，最多 30 跳
tracex 1.1.1.1 -c 5            # 跑 5 轮后退出
tracex baidu.com -m 20         # 限制最大跳数
tracex baidu.com -w 2          # 单跳等待 2s（链路慢时调大）
tracex baidu.com -W 30         # sparkline 窗口扩到 30 轮
tracex baidu.com --icmp        # 用 ICMP 探测（穿透防火墙更好，需 sudo）
```

`Ctrl-C` 退出。

颜色：

| 维度 | 规则 |
|------|------|
| 延迟 | 绿 < 50ms，黄 < 150ms，红 ≥ 150ms |
| 丢包 | 绿 0%，黄 < 20%，红 ≥ 20% |
| `*` 跳 | 灰色（路由器不响应 traceroute，但不一定意味着丢包，常见于跨骨干） |
| 目标 IP | 加粗（最后一跳到了目标） |

## 选项

| 选项 | 说明 | 默认 |
|------|------|------|
| `-m, --max-hops N` | 最大跳数              | 30 |
| `-w, --wait N`     | 单跳等待秒数          | 1  |
| `-c, --count N`    | 跑 N 轮后退出（0=无限）| 0  |
| `-W, --window N`   | sparkline 窗口         | 15 |
| `--icmp`           | ICMP 探测（需 sudo）   | 否 |

## 关于 `*` 跳

中间路由器经常配置成不返回 ICMP TIME_EXCEEDED（防止被探测），所以 traceroute 看到的中间一跳全 `*` 是**正常**的，不代表网络断了。重点看：

1. **目标那一跳**有没有数据 —— 若有，说明能通
2. **某一跳本来正常，突然开始丢/抖** —— 那一跳是问题源
3. **从某一跳开始所有后续都丢** —— 该跳之后链路出问题

## 关于 `--icmp`

默认用 UDP 探测（不需要权限）。`--icmp` 用 ICMP，穿透防火墙更好但 macOS 需要 sudo：

```bash
sudo tracex baidu.com --icmp
```
